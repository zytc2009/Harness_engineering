# 安装说明
> 给在新项目上部署 Harness Engineering 的工程师看。
> 这个文件是给你的，不是给智能体的。安装完成后删除此文件。

---

## 这套 Harness 解决什么问题

它让任何一个全新的智能体（没有任何对话历史）拿到五个文件、按顺序读完之后，就能立刻做到：

- 知道项目是什么，代码在哪里
- 把任务拆成若干阶段，按正确顺序推进
- 知道从哪里读输入，往哪里写输出
- 用统一的检查清单自我审查

Harness 不替代代码注释或 AGENTS.md。它替代的是你原本需要在每次新对话开始时和智能体进行的**定向对话**。

---

## 安装步骤

### 第一步 — 复制文件夹

```bash
cp -r harness-template/ <你的项目>/harness/
```

### 第二步 — 填写 HARNESS.md（最重要）

打开 `harness/HARNESS.md`，填写：

- **不变量**：本代码库 3–5 条不可妥协的规则（不可变性模型、并发模型、权限门控、输出约定、风格）
- **子系统地图**：每个主要模块一行，写入口文件和一行说明
- **关键类型**：5–10 个最重要的类型定义（伪代码即可）

**这是最重要的文件。** 一个只读了这一个文件的智能体，应该能在项目中找到方向而不迷路。

### 第三步 — 填写 BOOTSTRAP.md

打开 `harness/BOOTSTRAP.md`，填写：

- 项目简介（2–4 句话）
- 入口文件表（路径 → 职责）
- 目录地图
- 语言、风格、约定
- 开发命令（安装、构建、测试、冒烟测试）
- 「按任务类型决定先读什么」表
- 已知的坑

素材来源：现有的 AGENTS.md、CONTRIBUTING.md 或 README.md，大部分可以直接复制。

### 第四步 — 填写 TASK_PROTOCOL.md

打开 `harness/TASK_PROTOCOL.md`，填写**「特定任务类型的执行模式」**一节。

为项目中每种主要任务类型（新增工具、新增接口、新增数据模型等）写一个 5 步执行方法，告诉智能体该做什么、按什么顺序做。

四个通用阶段（理解→计划→实现→验证）保持不变，不要修改。

### 第五步 — 填写 IO_MAP.md

打开 `harness/IO_MAP.md`，填写：

- 配置来源（路径、格式、控制什么）
- 环境变量（名称 → 用途，标注密钥）
- 每种代码改动的写入位置
- 注册文件及添加方式
- 后台任务输出格式（如适用）
- 状态修改的规定模式 + 禁止的反模式
- 钩子 / 事件契约（如适用）

### 第六步 — 填写 REVIEW.md

打开 `harness/REVIEW.md`，填写：

- 冒烟测试命令（第 0 节）
- 注册完整性检查项（第 5 节）——每种能力类型一个检查块
- 并发 / 并行标志（第 6 节）——如果项目有这类注解
- 项目特定检查项（第 7 节）

### 第七步 — 删除本文件

```bash
rm harness/HOW_TO_INSTALL.md
```

### 第八步 — 告诉智能体从哪里开始

在项目的 AGENTS.md 或 README.md 中加一行：

```
> 全新智能体请先读 `harness/HARNESS.md`。
```

## 维护时机

以下情况需要更新 Harness：

| 发生了什么 | 更新哪个文件 |
|-----------|------------|
| 新增或重组了主要子系统 | HARNESS.md（子系统地图） |
| 新增了能力类型（工具类型、端点类型等） | TASK_PROTOCOL.md（执行模式）+ REVIEW.md（注册检查项） |
| 新增了配置来源或输出位置 | IO_MAP.md |
| 确立了新的全局规则或不变量 | HARNESS.md（不变量） |
| 开发命令或环境发生变化 | BOOTSTRAP.md |

**判断 Harness 是否已过时的标准**：如果一个智能体按照 Harness 指示写代码，结果写错了位置或漏掉了必须的注册步骤，那么 Harness 就需要更新了。

---

## Harness 不替代的东西

- `AGENTS.md` / `CONTRIBUTING.md` — 提交格式、PR 流程、构建细节
- 代码内注释 — 内联说明、JSDoc、docstring
- 架构决策记录（ADR）——重大决策的理由
- 测试文档 — 如何运行特定测试套件、测试数据准备
