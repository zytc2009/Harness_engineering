# 机械化检查规则 — 模板

> 文档会腐烂，lint 规则不会。
> 每条规则包含：检查方法 + 错误信息 + **修复指令**（agent 可直接执行）。
>
> 在中央层面强制执行边界，在本地层面允许自主权。

---

## 使用方式

sub-agent 在实现和测试之后、审查之前运行这些检查。
集成验证时对全部改动文件再跑一遍。

**任何 CRITICAL 检查失败 → 必须修复后才能继续。**
**WARNING 检查失败 → 应修复，但不阻塞。**

---

## CRITICAL 检查（阻塞）

### C1: 【检查名称】

```
检查：【如何检测，给出具体的 grep/命令/逻辑】

错误：{file}:{line} — 【错误描述】。
修复：【具体修复步骤，agent 可直接执行】。
      参考：【相关文档路径】。
```

---

## HIGH 检查（应修复）

### H1: 文件大小超限

```
检查：wc -l {file}
      【根据项目设置行数限制】

错误：{file} — {lines} 行，超过 {limit} 行限制。
修复：按职责拆分文件。
      参考：BOOTSTRAP.md §「语言、风格与约定」。
```

---

## WARNING 检查（建议修复）

### W1: TODO 残留

```
检查：grep -rn 'TODO\|FIXME\|HACK\|XXX' src/

错误：{file}:{line} — 残留 {tag}: {content}。
修复：解决 TODO 中描述的问题，然后移除注释。
```

---

## 添加新规则

当 LESSONS_LEARNED.md 中的模式错误被人类确认后，按以下格式添加新规则：

```
### {级别}{编号}: {简短描述}

\```
检查：{如何检测}

错误：{file}:{line} — {错误描述}。
修复：{具体修复步骤}。
      参考：{相关文档路径}。
\```
```

**关键：修复指令必须具体到 agent 可直接执行，不能只说"请修复"。**
