# Harness Engineering 模板使用说明

---

## 一、这套模板是什么

**Harness Engineering** 是一种实践：为项目搭建一套「智能体工作台」基础设施，让任何一个全新的 AI 智能体——哪怕没有任何历史对话上下文——都能直接接手工作，而不需要你先花时间给它讲解项目。

这套模板从 Claude Code（CC）项目的工程模式中提炼而来，抽象了以下核心机制：

- 任务生命周期管理（pending → running → completed/failed）
- 工具系统（接口定义 + 注册 + 权限门控 + 并发标志）
- 不可变状态管理（functional update 模式）
- 追加写入的磁盘输出（断点续读）
- 钩子驱动的生命周期自动化

---

## 二、模板文件一览

```
harness-template/
  使用说明.md            ← 你在读的这个文件
  HOW_TO_INSTALL.md      ← 安装步骤（安装完后删除）
  HARNESS.md             ← 智能体入口：决策树、不变量、子系统地图、关键类型
  BOOTSTRAP.md           ← 项目定向：是什么项目、目录结构、入口文件、开发命令
  TASK_PROTOCOL.md       ← 任务协议：四阶段拆解、执行顺序规则、类型特定模式
  IO_MAP.md              ← 输入输出地图：读哪里、写哪里、状态约定、钩子契约
  REVIEW.md              ← 审查清单：启动检查、正确性、安全、注册完整性、风格
```

---

## 三、智能体的工作流程

智能体收到任务后，按以下顺序工作：

```
读 HARNESS.md
  ↓ 用决策树确定任务落在哪个子系统
读 BOOTSTRAP.md
  ↓ 定位入口文件、目录结构、开发命令
读 TASK_PROTOCOL.md
  ↓ 阶段 A：理解（追踪执行路径，不猜测）
  ↓ 阶段 B：计划（写出改动列表，< 10 条）
  ↓ 阶段 C：实现（类型 → 逻辑 → 注册）
  ↓ 阶段 D：验证（冒烟测试 + 手动执行）
过 REVIEW.md 检查清单
  ↓ 全部通过后提交
```

## 四、安装到新项目的步骤

```bash
# 1. 复制模板
cp -r harness-template/ <你的项目>/harness/

# 2. 按顺序填写五个文件（见下方「填写优先级」）

# 3. 删除安装说明（它是给你的，不是给智能体的）
rm harness/HOW_TO_INSTALL.md

# 4. 在项目的 AGENTS.md 或 README.md 中加一行提示
echo "> 全新智能体请先读 \`harness/HARNESS.md\`。" >> AGENTS.md
```

## 五、填写优先级

按重要性从高到低：

| 优先级 | 文件 | 核心内容 | 参考素材 |
|--------|------|---------|---------|
| ★★★★★ | `HARNESS.md` | 不变量、子系统地图、关键类型 | 架构设计文档、技术债清单 |
| ★★★★☆ | `BOOTSTRAP.md` | 入口文件、目录结构、开发命令 | AGENTS.md、CONTRIBUTING.md、README |
| ★★★★☆ | `TASK_PROTOCOL.md` §「特定任务类型」 | 每种任务的 5 步执行方法 | 现有 PR 描述、架构会议记录 |
| ★★★☆☆ | `IO_MAP.md` | 配置路径、写入位置、状态约定 | 环境变量文档、部署说明 |
| ★★★☆☆ | `REVIEW.md` §5–7 | 注册检查项、并发标志、项目特定规则 | Code review 历史、Linter 配置 |

**最低可用版本**：只填 `HARNESS.md` + `BOOTSTRAP.md`，智能体就能完成简单任务。其余文件按需补充。

## 六、什么内容该写，什么不该写

### 该写进 Harness 的

| 类型 | 示例 |
|------|------|
| 硬约束 / 不变量 | 「状态永远不可变」「注册永远最后做」 |
| 结构性位置信息 | 「新工具写在 src/tools/<name>/ 下」 |
| 非显而易见的顺序依赖 | 「先改类型，编译器会告诉你影响范围」 |
| 容易踩的坑 | 「shims/ 目录是只读的，不要编辑」 |
| 项目特有的验证门槛 | 「`bun run version` 必须 exit 0」 |

### 不该写进 Harness 的

| 类型 | 原因 | 放哪里 |
|------|------|-------|
| 代码模式 / 架构 / 文件路径 | 从代码本身可以读出来 | 代码注释、JSDoc |
| Git 历史 / 谁改了什么 | `git log` 是权威来源 | — |
| 调试方法 / 修复方案 | 解决方案在代码里，上下文在提交信息里 | 提交信息、ADR |
| 已在 AGENTS.md 中的内容 | 重复 = 两处需要维护 | AGENTS.md |
| 当前任务的临时状态 | 跨对话后失效 | TodoWrite / 任务计划 |

## 七、维护规则

Harness 是「活文档」，需要随项目演化而更新：

```
新增子系统       → 更新 HARNESS.md 子系统地图
新增能力类型     → 更新 TASK_PROTOCOL.md 执行模式
                   更新 REVIEW.md 注册检查项
新增配置 / 输出  → 更新 IO_MAP.md
新增全局规则     → 更新 HARNESS.md 不变量
开发命令变化     → 更新 BOOTSTRAP.md
```

**判断是否需要更新**：智能体按 Harness 操作后，写错了位置或漏掉了必须步骤 → Harness 过时了，立即更新。

## 八、常见问题

**Q：这套 Harness 和 AGENTS.md 有什么区别？**

AGENTS.md 告诉智能体「这个项目的规则」（代码风格、提交格式、测试要求）。
Harness 告诉智能体「收到一个任务之后怎么工作」（怎么拆解、先读什么、往哪里写、怎么验证）。
两者互补，不重复。

**Q：五个文件都要填完才能用吗？**

不需要。最低可用版本是填完 `HARNESS.md` 和 `BOOTSTRAP.md`，覆盖 80% 的任务场景。其余文件随着项目复杂度增加逐步补充。

**Q：智能体真的会按这个顺序读吗？**

你需要在项目的 AGENTS.md 或者 system prompt 中明确指示：
「全新智能体请先读 `harness/HARNESS.md`。」
如果是通过 Claude Code 工作，也可以把这条写进 CLAUDE.md。

**Q：一个大型 monorepo 怎么用？**

根目录放一个 `harness/`（描述整体架构和不变量），每个子包再放一个 `harness/`（描述该包特有的入口文件和任务模式）。子包的 HARNESS.md 第一行写「先读根目录 harness/HARNESS.md」。

**Q：这套模板从哪里来的？**

从 Claude Code（CC）项目的源码中提炼，具体机制包括：Task 生命周期、Tool 接口 + 注册 + 并发标志、DeepImmutable AppState + setAppState 模式、TaskOutput 追加写入 + outputOffset 断点续读、Hook 系统的 stdin/stdout JSON 契约。这些机制被抽象成语言无关的通用模式。
