## 1. 正确性

- [ ] 改动后的代码是否满足任务要求？（重新读一遍原始需求。）
- [ ] 你是否手动执行了受影响的路径——不只是读代码？
- [ ] 你是否至少测试了一个成功案例和一个失败 / 边界案例？
- [ ] 如果任务有明确的验收标准，是否全部满足？

---

## 2. 不可变性 / 状态完整性

> 【根据项目的状态模型调整。下面给出示例。】

- [ ] 没有对共享状态对象进行原地修改
- [ ] 状态更新使用了项目规定的模式（见 `IO_MAP.md §「状态变更」`）
- [ ] 没有对状态数组执行 push / splice——使用展开运算符或不可变工具库
- [ ] 没有对状态中的 Map / Set 进行 mutation——创建新实例

---

## 3. 安全性

- [ ] 任何文件中都没有硬编码的密钥、API Key、Token 或密码
- [ ] 所有用户提供的字符串在到达文件系统、数据库或 Shell 之前都经过了校验
- [ ] 破坏性操作有明确的权限检查 / 确认门槛
- [ ] 错误信息不会向用户泄露内部路径、凭证或完整堆栈跟踪
- [ ] 用户输入没有被原始插值到 Shell 命令或 SQL 语句中

---

## 4. 错误处理

- [ ] 每个异步边界都有错误捕获——没有未处理的 Promise rejection
- [ ] 出错时，组件将自身状态设为终态失败（不是停留在"运行中"）
- [ ] 附属系统（钩子、插件、后台任务）失败时优雅降级——不会让主进程崩溃
- [ ] 用户看到的是有用的错误信息，不是原始堆栈跟踪或内部错误码

## 9. 收尾输出

所有检查通过后：

1. 输出简短总结：改了什么、为什么改、改动前后的行为差异。
2. 使用祈使句主题行提交：`修复 <X>`、`新增 <X>`、`重构 <X>`。
3. 在提交正文中记录任何权衡取舍、临时方案或后续待办事项。

提交格式和 PR 流程见 `BOOTSTRAP.md §「开发命令」` 及项目的 `AGENTS.md` / `CONTRIBUTING.md`。
