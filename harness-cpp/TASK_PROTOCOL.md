# 任务协议（TASK PROTOCOL）— 跨平台 C++20 项目
> C++ 项目的任务拆解与执行协议。

---

## 动手之前

1. 读 `HARNESS.md` — 确认你已知晓不变量和子系统地图。
2. 读 `BOOTSTRAP.md` — 确认你已知晓目录结构和 CMake 命令。
3. 确认你的**角色**，读对应的 `roles/<角色>.md`。
4. 判断任务涉及哪个子系统和哪些平台。
5. 只读负责该子系统的 2–3 个文件。不要通读整个代码库。

---

## 四个阶段（必须按此顺序执行）

### 阶段 A — 理解

目标：在改任何东西之前，彻底搞清楚代码当前的行为。

- [ ] 找到相关的类 / 接口定义（头文件优先）
- [ ] 从入口点追踪一条真实的调用链，直到要修改的代码
- [ ] 确认你将要改动的函数 / 类的所有调用方和被调用方
- [ ] 确认当前行为（读代码，不要猜测）
- [ ] **C++ 专属**：确认涉及的模板是否有显式实例化、特化，是否跨平台表现一致
- [ ] **C++ 专属**：确认改动是否涉及 ABI 变更（虚函数表布局、结构体内存布局、导出符号）

**如果任务描述与代码实际行为矛盾，在这里停下**，寻求澄清。

### 阶段 B — 计划

目标：动文件之前，先把要做的事情写下来。

- [ ] 列出所有需要修改的文件（头文件 + 实现文件 + CMakeLists.txt）
- [ ] 列出所有需要新增的类型 / 接口
- [ ] 识别必须保持不变的不变量（见 `HARNESS.md §「不变量」`）
- [ ] 判断影响范围：改动影响几个平台？是否需要在所有平台上验证？
- [ ] 确定实现顺序（见下方「执行顺序规则」）
- [ ] 如果涉及公共 API（`include/`），评估 ABI 兼容性影响

**把计划写成简短的要点列表。** 如果超过 10 条，说明任务太大——拆分它。

### 阶段 C — 实现

目标：满足任务要求的最小改动量。

按以下顺序执行（C++ 专属）：

1. **先改头文件（.h）** — 接口声明、类型定义、前向声明。
   编译器会在所有包含此头文件的编译单元报错，让你立刻看清全部影响范围。
2. **再改实现文件（.cpp）** — 函数实现、方法定义。
3. **然后改平台层** — 如果涉及平台特定代码，在 `src/platform/<平台>/` 中修改。
4. **再改测试** — 更新或新增对应的 `*_test.cpp`。
5. **最后改 CMakeLists.txt** — 新增源文件、新增目标、新增依赖。
   这一步永远是最后一步。

不要添加任务以外的功能。
不要重构你本不需要改动的代码。
不要为不可能发生的情况添加错误处理。

### 阶段 D — 验证

目标：在宣布完成之前，确认改动有效。

- [ ] 在当前平台运行构建 + 测试（`cmake --build + ctest`）
- [ ] 如果改动涉及多平台，确认其他平台的构建至少不会因语法/类型问题失败
- [ ] 手动执行受影响的路径至少一次
- [ ] 过一遍 `REVIEW.md` 检查清单

## 执行顺序规则

### 1. 头文件先于实现文件

先改 `.h`，编译器会成为你的依赖关系图。改动接口声明后，所有包含该头文件的 `.cpp` 都会报错，这就是你的影响范围清单。

### 2. 平台无关代码先于平台特定代码

先在 `src/core/` 中实现通用逻辑，然后在 `src/platform/` 中做平台适配。如果你发现自己在平台层写业务逻辑，说明抽象不够——退回去改接口。

### 3. 单个编译单元先于跨编译单元

尽量把改动限定在一个 `.cpp` 文件中。只有在编译器或测试强制要求时，才扩大范围。

### 4. 永远不要破坏公共 API

`include/` 目录下的头文件是公共契约。如果需要修改公共 API：
- 新增函数 / 类型：安全，直接添加
- 修改函数签名：必须评估 ABI 兼容性，考虑用重载而非修改
- 删除函数 / 类型：标记 `[[deprecated]]`，下个大版本再移除

### 5. CMakeLists.txt 最后改

新增源文件、目标或依赖的注册永远是最后一步。一个还没注册的源文件不会被编译（无害）；一个已注册但有编译错误的源文件会让整个构建失败。

## 特定任务类型的执行模式

### 模式：新增一个核心模块

```
1. 在 include/<项目名>/ 下创建公共头文件（如果需要对外暴露）
2. 在 src/core/ 下创建头文件 + 实现文件
3. 实现核心逻辑，遵循 RAII + Result<T,E> 错误处理
4. 在 tests/core/ 下创建 <模块名>_test.cpp，写 Google Test 用例
5. 在 src/core/CMakeLists.txt 中添加新源文件
6. 在 tests/CMakeLists.txt 中注册新测试目标
7. 验证：cmake --build + ctest 通过
```

### 模式：新增平台特定实现

```
1. 在 src/platform/platform.h 中定义或扩展抽象接口（纯虚函数）
2. 在每个平台目录下创建实现文件：
   - src/platform/windows/<功能>.cpp
   - src/platform/macos/<功能>.cpp（如需要 Obj-C++，用 .mm）
   - src/platform/android/<功能>.cpp
3. 实现接口方法，每个平台独立编译
4. 在 src/platform/CMakeLists.txt 中按平台条件添加源文件
5. 写平台无关的接口测试 + 各平台的集成测试
6. 验证：当前平台构建通过，其他平台至少头文件无编译错误
```

### 模式：新增第三方依赖

```
1. 在 vcpkg.json 中添加依赖声明（名称 + 版本约束）
2. 运行 vcpkg install 确认依赖可用
3. 在 CMakeLists.txt 中 find_package() + target_link_libraries()
4. 包装第三方 API（不让第三方头文件泄漏到 include/）
5. 验证：所有平台预设都能 configure + build 通过
```

### 模式：修复平台特定 Bug

```
1. 确认 Bug 发生在哪个平台（读 platforms/<平台>.md）
2. 在该平台的实现文件中定位问题代码
3. 修复时检查：同一逻辑在其他平台是否也有类似问题？
4. 写测试覆盖这个 Bug 场景
5. 验证：该平台 ctest 通过，其他平台不受影响
```

### 模式：Android JNI 桥接

```
1. 在 src/platform/android/jni/ 下定义 JNI 接口
2. JNI 函数仅做类型转换 + 调用 C++ 核心逻辑，保持薄层
3. 确保 JNI 函数中正确处理异常（C++ 异常不能跨 JNI 边界）
4. 在 CMakeLists.txt 中配置 SHARED 库目标（Android .so）
5. 验证：adb push + 运行，或通过 Android 测试框架
```

## 完成标准

满足以下全部条件，任务才算完成：

- [ ] 当前平台 `cmake --build + ctest` exit 0
- [ ] 手动执行受影响的路径，结果正确
- [ ] 没有新增编译器警告（`-Wall -Wextra -Wpedantic` / `/W4`）
- [ ] `REVIEW.md` 检查清单全部通过
- [ ] 输出写到了 `IO_MAP.md` 规定的正确位置
- [ ] 阶段 B 制定的计划已全部执行（没有跳过任何步骤）
- [ ] 如果改动影响多平台，至少确认过其他平台的编译兼容性
